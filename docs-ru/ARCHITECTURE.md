# Архитектура

Этот документ описывает высокоуровневую архитектуру NORA — мультипротокольного
реестра артефактов. Он предназначен для участников разработки, которые хотят
разобраться в кодовой базе, и для инженеров, оценивающих NORA для продакшена.

NORA — один бинарник на Rust (~34 тыс. строк), реализующий до 13 протоколов
реестров через один HTTP-порт. Это именно реестр — он предоставляет
протокол-совместимые интерфейсы для пакетных менеджеров (docker, npm, cargo,
pip и т.д.), а не хранилище данных. Нет базы данных, нет JVM, нет среды
исполнения расширений. Файловая система (или S3) — единственный источник истины.

## Принципы проектирования

1. **Один бинарник, ноль зависимостей.** Один бинарник `nora`, один файл
   конфигурации, один каталог данных. Никаких сайдкар-процессов, внешних баз
   данных, пакетных менеджеров в рантайме. `cp -r /data/ backup/` — полный бэкап.

2. **Файловая система — это база данных.** Всё состояние хранится на диске
   (или в S3) в виде файлов. Индексы в оперативной памяти пересоздаются
   при старте. Нет миграций схемы, нет риска повреждения WAL, нет команд
   `VACUUM`. Docker Distribution обслуживает Docker Hub тем же подходом.

3. **Безопасность бесплатна.** Чёрные списки, белые списки, изоляция
   пространств имён, проверка целостности — всё включено в открытую версию.
   Функции безопасности не должны быть платными дополнениями.

4. **Явное вместо абстрактного.** Каждый формат реестра имеет собственный
   модуль с явными маршрутами, явной конфигурацией, явными тестами. Нет
   типажей (trait) с виртуальной диспетчеризацией запросов в рантайме. Можно
   сделать `grep` по любой точке входа и найти ровно один обработчик.

5. **Форматы добавляются по запросу, не по контрольному списку.** Формат
   добавляется, когда реальные пользователи просят об этом, протокол хорошо
   специфицирован, и сопровождающие могут гарантировать: прокси работает,
   локальное размещение работает, тесты есть, документация есть.

## Архитектура системы

```
                           ┌─────────────────────┐
                           │    HTTP :4000        │
                           └──────────┬──────────┘
                                      │
                           ┌──────────▼──────────┐
                           │  Слой аутентификации │
                           │  (htpasswd + токены) │
                           └──────────┬──────────┘
                                      │
                           ┌──────────▼──────────┐
                           │ Ограничение частоты  │
                           │ (auth/upload/general)│
                           └──────────┬──────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          │                           │                           │
   ┌──────▼──────┐           ┌───────▼───────┐          ┌───────▼───────┐
   │   Docker    │           │     Maven     │   ...    │    Conan      │
   │  /v2/*      │           │  /maven/*     │ (из 13)  │   /conan/*    │
   └──────┬──────┘           └───────┬───────┘          └───────┬───────┘
          │                           │                           │
          └───────────────────────────┼───────────────────────────┘
                                      │
                           ┌──────────▼──────────┐
                           │  Механизм курации   │
                           │ чёрный→белый список  │
                           │ →пр-во имён→целостн. │
                           └──────────┬──────────┘
                                      │
                           ┌──────────▼──────────┐
                           │     Хранилище       │
                           │   local | s3        │
                           └─────────────────────┘
```

Каждый HTTP-запрос проходит этот путь сверху вниз. Обработчик реестра
выбирается по URL-префиксу (`/v2/` = Docker, `/maven/` = Maven и т.д.).
Курация выполняется только при проксировании скачиваний — локально
размещённые артефакты проверяются при публикации.

### Границы доверия

Пользовательский ввод поступает в систему на уровне обработчика реестра.
Каждый обработчик проверяет имена пакетов, версии и пути перед
формированием ключей хранилища. Модуль `validation.rs` предоставляет
`validate_storage_key()`, который отклоняет обход каталогов (path traversal),
нулевые байты и чрезмерно длинные ключи. Все обработчики вызывают эту
функцию перед любой операцией с хранилищем.

Слой курации — вторая граница доверия для проксируемого трафика. В режиме
`enforce` пакет должен пройти все фильтры (чёрный список, белый список,
пространство имён, целостность) прежде чем попадёт в хранилище. В режиме
`audit` заблокированные пакеты журналируются, но не отклоняются. Позиция
по умолчанию — запрет: если механизм курации ошибается при вычислении —
запрос блокируется, а не пропускается (fail-closed).

## Архитектурные решения

### ADR-1: Один бинарник

**Решение:** NORA поставляется как один статически слинкованный бинарник.
Все обработчики из 13 форматов реестров, веб-интерфейс, механизм курации
и инструменты командной строки скомпилированы в один исполняемый файл.

**Контекст:** Другие решения для реестров используют архитектуру расширений:
у Nexus — OSGi-модули, у Pulp — расширения на Python на каждый формат,
у Artifactory — Java-модули. Каждый подход создаёт управление зависимостями,
матрицы совместимости версий и ошибки загрузки в рантайме.

**Обоснование:** Один бинарник устраняет сложность развёртывания. Нет
расширений для установки, нет версий для согласования, нет
ClassNotFoundException. Бинарник занимает ~25 МБ в Docker-образе. Компромисс
в том, что неиспользуемые форматы занимают место — это компенсируется
возможностями Cargo (features) для исключения на этапе компиляции.

### ADR-2: Файловая система как источник истины

**Решение:** Всё состояние хранится как файлы на диске (или объекты в S3).
В открытой версии нет встроенной базы данных.

**Контекст:** Nexus мигрировал с файловой системы на OrientDB для метаданных.
Миграция заняла более двух лет и привнесла ошибки повреждения данных, которые
существуют до сих пор. SQLite предоставил бы структурированные запросы, но
добавляет второй источник истины, который может разойтись с реальными файлами.

**Обоснование:**
- `cp -r /data/ backup/` — полный, согласованный бэкап
- Нет миграций схемы, нет повреждения WAL, нет `VACUUM`
- Политика хранения использует mtime файла (дата публикации) — БД метаданных не нужна
- Поиск использует хеш-таблицу в памяти, пересоздаётся при старте (~5 мс для 10 тыс. пакетов)
- Токены хранятся в `tokens.json` — тот же подход, что htpasswd
- Docker Distribution обслуживает Docker Hub в масштабе на чистой файловой системе

### ADR-3: Два хранилища (локальная ФС + S3)

**Решение:** NORA поддерживает ровно два хранилища: локальную файловую систему
и S3-совместимое объектное хранилище. Третье добавляться не будет.

**Контекст:** Изначально рассматривался вариант использовать Nexus/Artifactory/GitLab
как хранилища, фактически превратив NORA в кэширующий прокси
перед другими реестрами.

**Обоснование:** Каждое хранилище — это поверхность сопровождения. S3 покрывает
всех облачных провайдеров и локальный MinIO. Локальная ФС покрывает
однонодовые установки и разработку. Третье хранилище (GCS-native, Azure Blob)
добавляет нагрузку на тестирование без существенного выигрыша — оба
S3-совместимы. Для миграции с других реестров утилита `nora migrate` копирует
артефакты напрямую, а не проксирует через старую систему.

### ADR-4: Явные обработчики вместо типажей-расширений

**Решение:** Каждый формат реестра — это отдельный модуль на Rust со своими
маршрутами, обработчиками, структурой конфигурации и тестами. Нет типажа
(trait) `RegistryPlugin` с диспетчеризацией в рантайме.

**Контекст:** Добавление нового формата реестра требует 24 точек вставки
в 9 файлах (см. «Добавление нового реестра» ниже). Участник разработки
отметил это как высокую связность (coupling).

**Обоснование:** Система расширений на типажах сократила бы количество
точек вставки, но привнесла бы новый слой абстракции: типаж
`RegistryPlugin` с ассоциированными типами, реализациями методов по
умолчанию и диспетчеризацией в рантайме. В Rust это означает
`Box<dyn RegistryPlugin>` или обобщения (generics), пронизывающие каждый
обработчик — оба варианта добавляют сложность, а не убирают её. У каждого
протокола реестра уникальная семантика (Docker — адресуемые по содержимому
блобы, Maven — контрольные суммы как файлы, Cargo — разрежённый индекс).
Общий типаж был бы либо слишком узким (требуя обходные пути для каждого
формата), либо слишком широким (протекая через десятки полей `Option<T>`).

Явный подход даёт практические преимущества:

- **Тестируемость.** Каждый обработчик — самостоятельный модуль со своим
  блоком тестов. Все 821 тестов выполняются за ~15 секунд через
  `cargo test`. Никакой загрузки расширений, имитаций типажей,
  интеграционных стендов.
- **Полнота на этапе компиляции.** При добавлении нового варианта
  `RegistryType` компилятор отмечает каждую необработанную ветвь
  сопоставления (match arm). Пропустить точку вставки — это ошибка
  сборки, а не сюрприз в рантайме.
- **Читаемость.** `grep "conan"` находит каждое место в кодовой базе,
  где упоминается Conan. Никакой косвенности через таблицы виртуальных
  методов (vtable) или объекты типажей (trait objects).

Новые форматы добавляются редко (6 добавлены в v0.7.0, следующие
ожидаются не раньше v0.9+). Стоимость 24 механических правок один раз
ниже, чем стоимость сопровождения слоя абстракции системы расширений
навсегда.

### ADR-5: Курация — сначала файлы, совместимо с GitOps

**Решение:** Правила курации (чёрные списки, белые списки) — это JSON-файлы
на диске. Их можно сгенерировать из файла зависимостей (lockfile) командой
`nora curation init --from-lockfile`. API для записи правил курации нет.

**Контекст:** Nexus Firewall хранит правила в своей БД. При повреждении
БД или случайном отключении функции все правила исчезают. В одном
задокументированном инциденте 588 пакетов просочились через отключённый
Nexus Defender.

**Обоснование:** Файловые правила хранятся в системе контроля версий,
просматриваются через `diff` и проходят рецензирование в запросах на
слияние (pull request). Механизм курации загружает правила в хеш-таблицу
в памяти для поиска за O(1). API — только для чтения (запрос решений).
Позиция «закрыто по умолчанию» (fail-closed) означает: если механизм
курации ошибается при вычислении — запрос блокируется, а не пропускается.

### ADR-6: Встроенный минимальный веб-интерфейс

**Решение:** Веб-интерфейс — серверный HTML, встроенный в бинарник. Он
только для чтения (просмотр реестров, пакетов, здоровья) с минимальным
управлением (API-токены).

**Контекст:** Участник разработки предложил вынести интерфейс в отдельное
одностраничное приложение (SPA) на React/Vue/Svelte.

**Обоснование:** 90% пользователей взаимодействуют с NORA через утилиты
командной строки (docker, npm, cargo, pip), а не через браузер. Встроенный
интерфейс обслуживает оставшиеся 10% — инженеров, проверяющих состояние
и просматривающих артефакты. Полноценное одностраничное приложение добавило
бы сборочный конвейер Node.js, настройку CORS и отдельный артефакт для
развёртывания. Текущий подход сохраняет эксплуатационные затраты на нуле:
интерфейс всегда доступен, всегда синхронизирован с API, не требует
отдельного процесса. Отдельное одностраничное приложение — вопрос для
рассмотрения в будущем.

### ADR-7: Динамическое подключение реестров

**Решение:** У каждого формата реестра — включая исходные 7 — есть
параметр `enabled` в конфигурации. Любой формат можно отключить.
Отключённые реестры не потребляют ресурсов — маршруты не монтируются,
фоновые задачи не запускаются.

**Контекст:** Из 13 доступных форматов большинству пользователей нужны
лишь 2-5. Монтирование всех маршрутов безусловно расходует память и
расширяет поверхность атаки.

**Обоснование:** Исходные 7 форматов (Docker, Maven, npm, Cargo, PyPI,
Go, Raw) включены по умолчанию для обратной совместимости. 6 новых
форматов (RubyGems, Terraform, Ansible, NuGet, Pub, Conan) отключены
по умолчанию. Допустима любая комбинация — можно запустить NORA только
с Docker и PyPI, выставив `NORA_MAVEN_ENABLED=false`,
`NORA_NPM_ENABLED=false` и т.д. Итератор `RegistryType::all()` и метод
`enabled_registries()` позволяют подсистемам (проверка здоровья, метрики,
интерфейс) автоматически обнаруживать активные форматы.

### ADR-8: Безопасность по умолчанию

**Решение:** Все функции безопасности (аутентификация, ограничение частоты
запросов, курация, изоляция пространств имён) включены в открытую версию
и активированы по умолчанию, где это безопасно.

**Контекст:** В других решениях для реестров безопасность — платное
дополнение: Artifactory требует Xray для сканирования CVE, Nexus требует
Firewall/Lifecycle для фильтрации пакетов. В итоге
безопасность скрыта за платной стеной.

**Обоснование:** Ограничение частоты запросов включено по умолчанию.
Аутентификация требует явного включения (файл htpasswd). Курация по
умолчанию выключена (`off`), но переключение на `enforce` — одно
изменение конфигурации. Изоляция пространств имён всегда активна при
настройке, независимо от режима курации. Цель: дефолтная конфигурация NORA безопаснее, чем дефолтная
конфигурация Nexus/Artifactory.

## Добавление нового реестра

Добавление нового формата требует 24 точек вставки в 9 файлах.
Полный список, прослеженный по обработчику Conan из v0.7.0:

| # | Файл | Точек | Что добавить |
|---|------|:-----:|-------------|

| 2 | `registry/mod.rs` | 2 | `mod <формат>;` и `pub use <формат>::routes as <формат>_routes;` |

## Известные компромиссы

**24 точки вставки на формат.** Добавление реестра требует 24 механических
правок в 9 файлах. Система расширений на типажах сократила бы это число,
но добавила бы слой абстракции, который нужно сопровождать вечно. Реестры
добавляются редко — явный подход обменивает разовый шаблонный код на
постоянную простоту, проверку полноты на этапе компиляции и полное покрытие
тестами каждого формата изолированно.

**Нет высокой доступности.** NORA работает как единственный экземпляр
с одним томом (RWO). Это проектное решение, а не упущенная возможность.
Реестры артефактов имеют нагрузку с преобладанием чтений — один экземпляр
с S3-хранилищем обрабатывает тысячи скачиваний в минуту. Стратегия
Kubernetes `Recreate` обеспечивает обновления без простоя для чтений,
обслуживаемых из клиентского кэша.

**Повторения между обработчиками.** Обработчики реестров разделяют
структурные шаблоны (логика проксирования, вызовы курации, загрузка
конфигурации), но различаются в деталях протокола. Дублирование реально.
Путь решения — макросы `macro_rules!` для шаблонного кода, а не
абстракция через типажи.

**Встроенный интерфейс минимален.** Серверный интерфейс покрывает
просмотр и мониторинг состояния, но не продвинутые операции (управление
пользователями, запросы к журналу аудита, визуальное редактирование правил
курации). Для этого лучше подходят внешние инструменты (панели Grafana,
управление правилами через git).

## Чем NORA НЕ является

- **Не CI/CD-система.** NORA — реестр: она предоставляет
  протокол-совместимый доступ к артефактам. Она не собирает, не тестирует
  и не развёртывает их.
- **Не сканер уязвимостей.** Курация блокирует заведомо опасные пакеты.
  Для сканирования CVE в ваших артефактах используйте Trivy, Grype
  или аналоги.
- **Не сборщик пакетов.** NORA не компилирует исходный код в пакеты.
  Используйте `cargo publish`, `npm publish`, `mvn deploy` для создания
  артефактов, затем отправляйте их в NORA.
- **Не CDN.** Для географически распределённой доставки артефактов
  поставьте CDN (CloudFront, Cloudflare) перед NORA.
- **Не промежуточный слой.** NORA — самостоятельный реестр, а не
  кэширующая прослойка перед Nexus или Artifactory. Для миграции
  используйте `nora migrate`.
